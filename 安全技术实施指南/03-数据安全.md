# 03 — 数据安全（Data Security）
**对应 MAS TRM §13 | PDPA | PCI DSS v4.0**
当前状态：部分已实施

---

## 1. 数据分类标准

**第一级 — 限制访问（最高敏感）**
→ 卡号（PAN）、CVV、PIN — PCI DSS 管控范围
→ 身份证号、护照号
→ 银行账号、IBAN
→ 私钥、HSM 种子
→ 处理要求：字段级加密 + 访问需审批 + 完整审计日志

**第二级 — 机密（Confidential）**
→ 姓名 + 出生日期 + 地址（PII 组合）
→ 手机号、邮箱
→ KYC 文件（护照照片、身份证照片）
→ 交易记录、VA 余额
→ 处理要求：传输加密 + 存储加密 + 角色访问控制

**第三级 — 内部（Internal）**
→ 用户 ID、账户 ID（非敏感标识符）
→ 脱敏后的交易摘要
→ 不含 PII 的系统日志
→ 处理要求：传输加密，存储可不加密

**第四级 — 公开（Public）**
→ 产品介绍、公开 API 文档
→ 汇率信息、费率表

## 2. 数据库加密规范

### 2.1 RDS PostgreSQL — 必须加密的字段

**存储层加密（静态加密）**
→ RDS 开启 AWS KMS AES-256 全盘加密 ✅ 已实施
→ 所有快照自动加密

**字段级加密（必须项）— 以下字段必须在应用层加密后再写入数据库：**

```sql
-- users 表
id_number         TEXT  -- 加密存储（身份证/护照号）
date_of_birth     TEXT  -- 加密存储
phone_number      TEXT  -- 加密存储
email             TEXT  -- 加密存储（搜索用 HMAC 哈希索引）
full_address      TEXT  -- 加密存储

-- cards 表
pan_encrypted     TEXT  -- 加密存储，绝对不允许明文存储
pan_last4         CHAR(4)  -- 允许明文（仅用于界面展示）
pan_token         TEXT     -- 支付 Token（替代明文 PAN 在系统内流转）
-- CVV：永不存储，任何表、任何形式都不允许存储 CVV

-- kyc_verifications 表
sumsub_applicant_id  TEXT  -- 加密存储
raw_response         TEXT  -- 加密存储（含完整 KYC 结果）
```

```java
        // 第四步：立即从内存清除明文数据密钥
    }

    public String 解密(String 密文) {
        // 第一步：拆分加密数据密钥和密文
        // 第二步：调用 KMS 解密数据密钥
        // 第三步：用数据密钥解密密文
        // 第四步：立即从内存清除明文数据密钥
```

**PAN Tokenization（PCI DSS §3 要求）：**
```
真实 PAN：4111 1111 1111 1111
Token：   7823 4521 9876 0001（不可逆 Token，在系统内流转用）
```

### 2.2 Redis / ElastiCache

**强制规则：**
→ 开启传输 TLS 1.2+ ✅ 已实施
→ 开启静态加密（ElastiCache）
→ Redis 中禁止存储完整 PAN、CVV、身份证号
→ Session Token 存储 TTL 最长 24 小时
→ Cache Key 不包含可识别的用户 ID（使用哈希值）

```
❌ 错误示范：SET user:12345:profile {"pan":"4111...","cvv":"123"}
✅ 正确示范：SET session:sha256(token):{} TTL=3600
```

### 2.3 S3 — KYC 文件存储

**必须配置项：**
```
存储桶策略：
  → 封锁所有公共访问：开启
  → 服务端加密：SSE-KMS（AES-256）
  → 版本控制：开启
  → MFA 删除保护：开启（防止误删或恶意删除）
  → 访问日志：开启 → 写入独立的审计存储桶

访问控制：
  → 仅允许 KYC 服务的 IAM Role 读写
  → 前端用户不直接访问 S3
  → 生成 Presigned URL，有效期最长 15 分钟
  → URL 使用后立即失效（通过 CloudFront + Lambda@Edge 实现一次性访问）

保留策略：
  → KYC 文件保留 5 年（MAS/PDPA 要求）
  → 5 年后自动归档至 Glacier → 7 年后自动删除
```

## 3. 传输加密规范

### 对外 API（客户端 ↔ 服务器）
```
协议：TLS 1.2 最低，推荐 TLS 1.3
禁用：TLS 1.0、TLS 1.1、SSL 3.0
HSTS：max-age=63072000; includeSubDomains; preload
证书：AWS ACM 管理，自动续期
```

### 内部服务间（服务 ↔ 服务）
```
方案：Istio mTLS（STRICT 强制模式）
证书：Istio CA 自动签发，90 天自动轮换
验证：双向证书验证，服务身份无法伪造
```

### 数据库连接
```
PostgreSQL：sslmode=verify-full（客户端必须验证服务器证书）
Redis：TLS=true，cert-reqs=required
Kafka：security.protocol=SSL + SASL_SSL
```

## 4. 数据保留与销毁（PDPA 合规）

```
KYC 文档：         保留 5 年（账户注销后起算）→ 之后安全销毁
交易记录：         保留 5 年（MAS PSA 要求）
应用日志：         保留 1 年（含 PII 的日志必须脱敏后保留）
安全审计日志：     保留 3 年（MAS TRM 要求）
STR 可疑交易报告： 保留 5 年（MAS Notice PSN01）
```

**数据销毁要求：**
→ S3 数据：AWS 安全删除（覆盖写入）
→ RDS 数据：应用层先覆盖字段值，再执行 DELETE
→ 硬件销毁：需要符合 DoD 5220.22-M 标准的销毁证书

## 6. 密钥管理架构

```
第一层：AWS KMS（主密钥）
→ 永不离开 AWS HSM
→ 用于加密数据密钥

第二层：AWS KMS 数据密钥
→ 每次加密操作生成独立数据密钥
→ 明文数据密钥使用后立即销毁

第三层：HashiCorp Vault
→ 存储应用配置密钥、API Key、数据库密码
→ 动态凭证（数据库密码每次请求都不同，用完即废）
→ 所有密钥访问有完整审计日志

第四层：区块链私钥
→ 存储在 Fireblocks HSM
→ MPC 多方计算签名
→ 无单点持有，任何单一方都无法独立控制私钥
```
