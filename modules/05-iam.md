# Module 05: Identity & Access Management
**MAS TRM §10 — Access Control**
Status: Partially Implemented

---

## 1. Role Hierarchy

```
Super Admin (CISO / CTO only)
  ├── Production Admin     → Full prod access, all changes require ticket
  ├── Security Admin       → Security tools, WAF, SIEM config
  └── Compliance Read      → Read-only audit access

Team Lead
  ├── Developer            → Dev/Staging only, NO prod access by default
  ├── DevOps / SRE         → Prod read + deploy via CI/CD pipeline only
  └── Operations           → Prod read + specific operational commands

Read Only
  └── Analyst / Support    → Masked data, no PII visible
```

---

## 2. MFA Requirements (Mandatory)

```
MUST have MFA:
  ✅ All AWS Console access
  ✅ All production system access (K8s, databases)
  ✅ VPN / ZTNA (SealSuite)
  ✅ All privileged accounts (admin/root)
  ✅ Code repository (GitHub) for production branches
  ✅ SIEM and security tools
  ✅ Any remote access

Acceptable MFA methods:
  → Hardware FIDO2 key (YubiKey) — REQUIRED for CISO/CTO/Prod Admin
  → TOTP authenticator app (Google Authenticator / Authy) — minimum standard
  → SMS OTP — NOT acceptable for privileged accounts (SIM swap risk)
```

## 3. API Authentication

**External APIs (/public/ namespace):**
```
Method: OAuth 2.0 + JWT
Token lifetime: Access token 15 min, Refresh token 7 days
Token storage: httpOnly cookie (not localStorage)
Scope enforcement: per-endpoint scope validation
```

**Internal APIs (/internal/ namespace):**
```
Method: Internal Service Token (X-Internal-Token header)
        + Source IP validation (must be within private subnet CIDR)
Token: Generated by Vault, rotated every 24h
Never exposed to public internet
```

**Service-to-Service:**
```
Method: mTLS (via Istio)
Certificate: Auto-issued by Istio CA, 90-day rotation
Identity: SPIFFE/SPIRE service identity (not just IP)
```

**Admin / Management APIs:**
```
Method: OAuth 2.0 + MFA step-up for sensitive operations
All admin actions logged with requester identity
```

## 4. Privileged Access Management (PAM)

**Principles:**
→ No shared admin credentials — every admin has individual identity
→ No persistent privileged sessions — just-in-time (JIT) access
→ All privileged sessions recorded (terminal recording to S3)
→ Break-glass accounts for emergency only, requires two approvers

**Implementation (HashiCorp Vault):**
```
Database credentials:
  → Dynamic credentials: Vault generates a unique username/password per request
  → Credential lifetime: 1 hour for developers, 15 min for automated pipelines
  → Auto-revoked on session end

SSH / Server access:
  → Vault SSH CA signs temporary certificates
  → Certificate valid for 1 hour only
  → No permanent SSH keys on servers

Secrets management:
  → All API keys, DB passwords, certificates stored in Vault
  → Application retrieves secrets at startup via Vault agent
  → No secrets in code, config files, or environment variables in CI/CD logs
```

## 5. AWS IAM — Least Privilege

```
Rules:
  → No wildcard (*) permissions in production IAM policies
  → Separate AWS accounts for: dev / staging / production
  → No human access with long-lived AWS access keys
    (use IAM roles + instance profiles / OIDC federation)
  → All IAM changes go through Infrastructure as Code (Terraform)
  → MFA required for all IAM console access

Required boundary policies:
  → Deny: CreateAccessKey (prevent creating long-lived keys)
  → Deny: All actions if MFA not present (for human roles)
  → Deny: Actions outside ap-southeast-1 (Singapore region lock)
```

## 6. Access Review Schedule (MAS TRM §10)

```
Frequency: Quarterly (every 3 months)
Scope:
  → All production system access rights
  → Privileged account list
  → Service account inventory
  → AWS IAM role assignments

Process:
  → System owners review and certify their user lists
  → CISO approves all privileged access
  → Orphaned accounts (no activity 90 days) → automatic disable
  → Terminated employee accounts → disable within 4 hours of HR notification
```

---

## 7. Offboarding Checklist (Same-Day Execution)

```
Trigger: HR notifies Security team of termination

Within 4 hours:
  □ Disable SSO / corporate account
  □ Revoke VPN / SealSuite ZTNA access
  □ Revoke GitHub access
  □ Rotate any shared secrets the employee had access to
  □ Review their recent access logs (last 30 days)
  □ Disable AWS IAM access
  □ Deactivate all API keys issued to them

Within 24 hours:
  □ Recovery of corporate devices
  □ Remote wipe if device not returned
  □ Archive mailbox
  □ Confirm no active sessions
```
